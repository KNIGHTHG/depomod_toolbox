Navigating a project using Matlab

This guide explains how to use Matlab to navigate the files of a project and access required information.

Project file structure

Existing projects represent directories of files on the file system which conform to the expected NewDepomod structure. The file structure is roughly as follows (described relative to the project root directory)
- \bathymetry
- bay_of_fish.depomodbathymetryproperties
(domain/bathymetry/land data)
- \cages
- bay_of_fish-1.depomodcagesxml
- bay_of_fish-2.depomodcagesxml
- etc...
(cage definitions for every run)
- \flowmetry
- bay_of_fish.depomodflowmetryproperties
(flow data)
- \inputs
- bay_of_fish-NONE-1-allCages.depomodinputsproperties
- bay_of_fish-NONE-2-allCages.depomodinputsproperties
- bay_of_fish-EMBZ-1-allCages.depomodinputsproperties
- bay_of_fish-EMBZ-2-allCages.depomodinputsproperties
- etc...
(discharge definitions for every run (feed inputs, etc.))
- \models
- bay_of_fish-1-NONE.depomodconfigurationproperties
- bay_of_fish-1-NONE.depomodmodelproperties
- bay_of_fish-1-NONE.depomodphysicalproperties
- bay_of_fish-1-NONE.depomodruntimeproperties
- bay_of_fish.depomodlocationproperties
- bay_of_fish-1-EMBZ.depomodconfigurationproperties
- bay_of_fish-1-EMBZ.depomodmodelproperties
- bay_of_fish-1-EMBZ.depomodphysicalproperties
- bay_of_fish-1-EMBZ.depomodruntimeproperties
- etc...
(set up definitions for every run)
- \results
- bay_of_fish-1-NONE-N.depomodresultslog
- bay_of_fish-1-NONE-N-carbon-g0.depomodresultssur
- bay_of_fish-1-NONE-N-carbon-g1.depomodresultssur
- etc...
(results files for optimised runs; only exist following successful run completion)
This project file structure is represented by NewDepomod.Project objects in order to aid the navigation and manipulation of the project files. This allows files to be accessed without explicitly handling their filenames or paths - to code understands where each file resides.
The examples below assume that a project object has been initialised.

Retrieve project information

The project object can be queried in a variety of ways. For example, we can call up the project name.
project.name
ans = 'bay_of_fish'
We can return the project location.
project.path
ans = 'C:\newdepomod_projects\bay_of_fish'
or the parent directory of the project.
project.parentPath
ans = 'C:\newdepomod_projects'
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\flowmetry'
ans = 1
Or we can return the centre of the model domain represented in the project
[e,n] = project.centre
e = 350340
n = 1068960

Retrieve project subdirectories

project.bathymetryPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\bathymetry'
project.flowmetryPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\flowmetry'
project.inputsPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\inputs'
project.modelsPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\models'
project.resultsPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\results'

Retrieve individual filepaths

Individual files can also be discovered. For example
project.bathymetryDataPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\bathymetry\bay_of_fish.depomodbathymetryproperties'
project.flowmetryFilePath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\flowmetry\bay_of_fish.depomodflowmetryproperties'
project.locationPropertiesPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\models\bay_of_fish.depomodlocationproperties'
And we can drill down in to individual model runs to discover the associated files
project.solidsRuns.first.cagesPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\cages\bay_of_fish-1.depomodcagesxml'
project.solidsRuns.first.inputsFilePath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\inputs\bay_of_fish-1-NONE-allCages.depomodinputsproperties'
project.solidsRuns.first.physicalPropertiesPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\models\bay_of_fish-1-NONE.depomodphysicalproperties'
project.solidsRuns.first.modelPath
ans = 'C:\newdepomod_projects\bay_of_fish\depomod\models\bay_of_fish-1-NONE.depomodmodelproperties'
Note that the .first function is a special case of the .item() function which can be invoked on either solids or EmBZ run collections. Other functions can be used to identify specific model runs depending on the labelling systems used (.number(), .label()), though these are described in more detail in other guides.
