Creating a project

This guide explains how to create a new project.

Creating projects

The library contains a template project which is used to produce the skeleton file structure for a new project.
The template includes sample bathymetry, cage and flow data and is initialised with a single solids run configured with a biomass of 1000 t and a single EmBZ run configured with a treatment quantity of 350 g.
In the general case this method can be used to create a new project after which the bathymetry, cage and flow files should be edited to reflect the required scenario and the initial run files edited to reflect the desired modelling starting point. Additional runs can then be added easily using this library.

Define project name and location

The first thing to do when creating a new project is to define a file system location and a project name. The location will become the parent directory of the project, and the project name will be used for the project directory and as a prefix on most project files. There is no constraint on where projects should be stored, but a useful location is described below;
rootDir = 'C:\newdepomod_projects\';
if ~exist(rootDir, 'dir')
mkdir(rootDir);
end
projectName = 'bay_of_fish';

Create project

Next, a new project can be created according to the specification given.
project = NewDepomod.Project.createFromTemplate(rootDir, projectName)

This creates a new project complete will all requires files. It also returnd a Matlab object which represents the project. This object can be used to navigate around the project files, reading, editing and creating new files as required.

Force overwrite

Note that the .createFromTemplate() function will not run if a project directory corresponding to the name and location specified already exists. This is intended to prevent the accidental overwriting of a modelling project.
This behaviour can be overruled by using the force option, e.g.
project = NewDepomod.Project.createFromTemplate(rootDir, projectName, 'force', 1)
in which case any existing file will be deleted and replaced by the new project. This means previous results file may be lost. Care should be taken, therefore, when using this option.

Important note

This Matlab library makes a number of assumptions about how users expect to organise their model files. These assumptions are not enforced and can be changed, but they should be understood.
NewDepomod organised simulations according to cage layouts. These cage layouts require unique descriptors which can be alpha-numeric. Some of the functionality in this library makes it most straightforward to organise simulations around a simple, sequentially numbered labelling system - i.e. runs 1, 2, 3, 4... etc. In cases where more descriptive alpha numeric labelling of runs is preferred the library can handle this, but it may take a greater level of attention on the part of the user.
Secondly, this library makes the assumption that every simulation uses its own cohort of configuration files. This means that no configuration files are shared across runs, whihc is the default behaviour of the user interface. Projects created using this library are therefore configured to reference individual configurations files for each run. Again, this behaviour can be overridden, but it should be understood.
